# Trading Studio 数据质量规范

## 1. 目的

本文档用于定义 Trading Studio 在证券主数据、行情、新闻、公告、事件和 AI 输出方面的数据质量标准、监控指标、处理流程和验收门槛。

## 2. 质量目标

- 保证核心数据可用、可信、及时、可追溯
- 让上游数据源异常不会直接污染前台和 AI 输出
- 为数据补录、修正、告警和回归测试提供统一标准

## 3. 质量维度

| 维度 | 定义 | 典型检查项 |
| --- | --- | --- |
| 完整性 | 必填字段是否齐全 | 股票代码、发布时间、标题、收盘价 |
| 准确性 | 值是否与事实一致 | 涨跌幅、公告时间、实体关联 |
| 一致性 | 同一实体跨表是否一致 | 股票标识、板块映射、时间格式 |
| 唯一性 | 重复数据是否被控制 | 新闻去重、事件聚类、日线唯一键 |
| 时效性 | 数据是否按目标窗口到达 | 快讯延迟、公告延迟、行情更新时间 |
| 可追溯性 | 是否能追溯到原始载荷和解析版本 | raw payload、checksum、parser_version |
| 合规性 | 数据是否符合授权和版权边界 | 原文存储方式、转载限制 |

## 4. 数据分层

### 4.1 原始层 Raw

- 保存源站原始响应和抓取元数据
- 不对外直接使用
- 必须保留 `request_id`、`checksum`、`raw_payload_path`

### 4.2 标准层 Standardized

- 将不同来源映射到统一字段模型
- 完成时间格式、代码格式、数值字段归一化

### 4.3 主题层 Domain

- 形成新闻、公告、行情、事件、AI 任务等业务实体
- 对外接口和页面优先依赖此层

### 4.4 分析层 Analytics

- 存储聚类结果、事件影响、历史回测、报表指标
- 可用于运营看板和后续模型训练

## 5. 通用数据质量元字段

每条重要记录都应尽量具备以下元字段：

- `provider`
- `source_url`
- `source_item_id`
- `fetched_at`
- `source_timestamp`
- `raw_payload_path`
- `parser_version`
- `quality_status`
- `quality_score`
- `error_message`

## 6. 数据集质量规则

### 6.1 证券主数据

必检规则：

- `canonical_symbol` 必填且唯一
- `symbol + exchange` 唯一
- `security_type` 必须在枚举内
- `list_date <= delist_date` 或 `delist_date` 为空
- 停牌、退市状态需可追踪

告警阈值：

- 新增证券重复率 > 0.1% 告警
- 核心字段空值率 > 1% 告警

### 6.2 日行情与快照

必检规则：

- `trade_date` 必须是交易日
- `high >= max(open, close, low)`
- `low <= min(open, close, high)`
- `volume >= 0` 且 `amount >= 0`
- 同一股票同一交易日同一复权类型只能有一条

告警阈值：

- 行情缺失股票数 > 0.5% 告警
- 单日全市场记录数偏离 3 日均值超过 20% 告警

### 6.3 新闻数据

必检规则：

- 标题、发布时间、来源、正文至少满足基础字段要求
- 正文长度低于最小阈值时标记低质量
- 新闻必须经过去重和聚类链路

告警阈值：

- 正文抽取失败率 > 5% 告警
- 同源重复率 > 10% 触发检查

### 6.4 公告数据

必检规则：

- 公告类型必须可识别
- 股票关联不能为空
- 原文链接或 PDF 链接至少保留一项
- 摘要需标注提取时间和 Prompt 版本

告警阈值：

- 公告抓取延迟超过 10 分钟告警
- 公告解析失败率 > 3% 告警

### 6.5 事件数据

必检规则：

- `event_type`、`occurred_at`、`importance_level` 必填
- 事件必须至少关联一个实体
- 同类同主体同时间窗口内应尝试合并
- `confidence` 必须在 `0 ~ 1` 区间

告警阈值：

- 无主体事件比例 > 2% 告警
- 事件合并冲突率 > 5% 触发抽检

### 6.6 AI 输出

必检规则：

- 结构化任务必须通过 schema 校验
- 输出必须包含引用和风险声明
- 未标注数据时间的回答不得前台展示
- 出现禁用词或收益承诺直接拦截

告警阈值：

- Schema 失败率 > 2% 告警
- 引用缺失率 > 5% 告警
- 合规拦截率异常升高需人工复核

## 7. 质量检查层级

### 7.1 同步前检查

- Source 配置是否启用
- 凭证和访问参数是否有效
- 当日交易日历和任务窗口是否正确

### 7.2 入库前检查

- 必填字段校验
- 枚举和值域校验
- 唯一键冲突处理
- 时间与数字格式标准化

### 7.3 入库后检查

- 记录数对比
- 空值比例
- 样本抽检
- 跨表一致性检查

### 7.4 对外发布前检查

- AI 输出合规校验
- 重大事件人工复核
- 前台缓存刷新验证

## 8. 质量评分模型

建议对单批数据记录质量评分：

```text
quality_score =
  completeness * 0.30 +
  accuracy * 0.25 +
  timeliness * 0.20 +
  consistency * 0.15 +
  uniqueness * 0.10
```

建议状态：

- `pass`：`score >= 0.90`
- `warning`：`0.75 <= score < 0.90`
- `failed`：`score < 0.75`

## 9. 告警分级

| 级别 | 说明 | 处理方式 |
| --- | --- | --- |
| `P1` | 影响前台核心数据或 AI 事实依据 | 立即通知并暂停发布 |
| `P2` | 影响部分来源或部分模块 | 通知值班人员并限时修复 |
| `P3` | 轻微异常或趋势偏移 | 进入观察和工单池 |

## 10. 异常处理流程

1. 任务自动标记失败并写入 `data_job_runs`
2. 记录错误样本、请求参数、栈信息和原始载荷位置
3. 若达到阈值则触发熔断和告警
4. 由值班人员判断是数据源异常、解析逻辑异常还是字段契约变化
5. 修复后执行补跑和回归测试
6. 对前台和 AI 已消费的数据做影响评估

## 11. 回归测试要求

以下情况必须执行数据质量回归：

- 升级 AKShare 或其他 Provider 版本
- 修改 parser、字段映射或实体识别逻辑
- 调整事件抽取 Prompt 或 schema
- 新增新闻源或公告源

回归至少覆盖：

- 10 条历史稳定样本
- 10 条异常样本
- 5 条边界样本
- 关键指标变化对比

## 12. 监控指标

### 12.1 数据指标

- 数据源成功率
- 数据源平均延迟
- 每日采集量
- 去重率
- 事件生成量
- 事件人工通过率

### 12.2 质量指标

- 空值率
- 重复率
- Schema 通过率
- 解析失败率
- 聚类命中率
- AI 引用覆盖率

## 13. 数据保留策略

| 数据类型 | 建议保留期 |
| --- | --- |
| 原始抓取载荷 | 90 到 180 天 |
| 新闻正文清洗结果 | 长期保留 |
| AI 输入输出日志 | 至少 180 天 |
| Provider 健康日志 | 至少 180 天 |
| 失败任务日志 | 至少 180 天 |

## 14. 发布门禁

满足以下条件才允许上线：

- 核心数据集质量评分达标
- 告警未积压 `P1`
- 回归测试通过
- 核心前台页面抽样核验通过
- AI 输出抽检通过
