# Trading Studio 部署运维手册

## 1. 目标

本文档用于指导 Trading Studio 在开发、测试和生产环境中的部署、发布、监控、备份、恢复和应急处理。

## 2. 系统组成

### 2.1 核心组件

- `web`：Nuxt 4 前端
- `laravel`：Laravel 10 业务后端
- `laravel-queue`：Laravel 队列消费
- `laravel-scheduler`：Laravel 定时任务
- `fastapi`：数据与 AI API
- `worker-market`：行情与主数据 Worker
- `worker-news`：新闻与公告 Worker
- `worker-ai`：AI 任务 Worker
- `scheduler-python`：Python 定时调度
- `mysql`：MySQL 8
- `redis`：缓存、队列、锁
- `qdrant`：向量数据库
- `object-storage`：对象存储或兼容 S3 服务
- `nginx`：反向代理、静态资源、WAF 和限流

## 3. 环境规划

| 环境 | 用途 | 说明 |
| --- | --- | --- |
| `dev` | 本地开发 | Docker Compose，允许调试工具 |
| `staging` | 集成测试与验收 | 与生产接近，接入测试数据源 |
| `prod` | 生产环境 | 高可用、备份、告警和访问控制 |

## 4. MVP 部署拓扑

### 4.1 推荐拓扑

```text
服务器 A
  - Nginx
  - Nuxt
  - Laravel PHP-FPM
  - Laravel Queue
  - MySQL
  - Redis

服务器 B
  - FastAPI
  - Worker Market
  - Worker News
  - Worker AI
  - Scheduler Python
  - Qdrant
```

### 4.2 最低资源建议

- 业务机：4 核 8 GB
- Python 数据机：8 核 16 GB
- SSD：200 GB 起
- 数据库和对象存储单独备份

## 7. Docker Compose 服务建议

```yaml
services:
  nginx:
  web:
  laravel:
  laravel-queue:
  laravel-scheduler:
  fastapi:
  worker-market:
  worker-news:
  worker-ai:
  scheduler-python:
  mysql:
  redis:
  qdrant:
```

说明：

- `laravel` 与 `laravel-queue` 镜像可共用
- `worker-market`、`worker-news`、`worker-ai` 可根据负载分别扩容
- 生产环境不建议把数据库和应用放在同一 Compose 文件长期耦合

## 8. 发布流程

### 8.1 预发布检查

- 核心环境变量已配置
- 数据库迁移脚本已审查
- OpenAPI 变更已同步
- Prompt 和模型配置已在后台预置
- 监控与告警已启用

### 8.2 标准发布步骤

1. 拉取发布版本代码
2. 构建前端、Laravel、FastAPI 镜像
3. 执行数据库迁移
4. 发布静态资源
5. 逐个拉起服务
6. 执行健康检查和烟雾测试
7. 观察日志、错误率、队列和数据任务状态

### 8.3 建议顺序

1. 数据库迁移
2. FastAPI
3. Worker 与 Scheduler
4. Laravel
5. Nuxt
6. Nginx reload

## 9. 健康检查

### 9.1 必备检查项

- `GET /api/v1/health` 或等价业务健康页
- `GET /internal/v1/health/providers`
- MySQL 连接
- Redis 连接
- Qdrant 连接
- 对象存储读写
- 队列积压

### 9.2 发布后检查

- 用户登录
- 股票搜索
- 新闻列表
- 公告详情
- AI 任务创建与轮询
- 自选股添加

## 10. 监控与日志

### 10.1 监控指标

- API 成功率、延迟、QPS
- 队列长度和消费延迟
- 新闻采集量、公告采集延迟
- AI 任务成功率、首 Token 延迟、成本
- MySQL 慢查询
- Redis 内存与连接数

### 10.2 推荐工具

- Prometheus
- Grafana
- Loki
- Sentry
- OpenTelemetry
- Laravel Horizon

### 10.3 日志要求

- 日志必须结构化输出
- 日志中保留 `request_id`、`task_id`、`provider`
- 不记录密码、Token、API Key 和完整敏感响应

## 11. 备份策略

### 11.1 数据库

- MySQL 每日全量备份
- 关键业务表每小时 binlog 或增量备份
- 备份至少保留 7 天本地副本和 30 天远端副本

### 11.2 对象存储

- 原始公告、抓取载荷、导出报告启用版本管理
- 每日校验对象存储桶可用性

### 11.3 配置

- `.env` 不直接备份到公开位置
- 所有密钥使用 Secret Manager 或受控加密存储

## 12. 恢复流程

### 12.1 数据库恢复

1. 停止写入流量
2. 恢复最近一次全量备份
3. 回放增量日志
4. 核验用户、订单、事件和 AI 任务关键表
5. 启动只读检查
6. 恢复服务流量

### 12.2 应用恢复

- 通过上一个稳定镜像版本回滚
- 若迁移不可逆，需准备兼容迁移或恢复脚本

## 13. 扩容建议

### 13.1 应用层

- Nuxt 和 Laravel 可横向扩容
- FastAPI API 与 Worker 分开扩容
- AI Worker 与批处理 Worker 必须隔离队列

### 13.2 数据层

- MySQL 后续可升级主从或云数据库
- Redis 升级 Sentinel 或 Cluster
- Qdrant 根据向量量级扩为集群

## 14. 安全要求

- FastAPI 不直接暴露公网
- MySQL 和 Redis 禁止公网访问
- 全部服务走 HTTPS
- 生产容器使用非 root 用户
- 镜像在发布前执行漏洞扫描
- 内部接口启用 HMAC、时间戳和 nonce

## 15. 定时任务建议

### 15.1 交易日任务

- 盘前：更新交易日历、指数概览、自选股简报
- 盘中：新闻、公告、快讯、快照行情
- 盘后：事件计算、盘后复盘、AI 摘要

### 15.2 非交易时段

- 历史补数
- 向量重建
- 数据质量回归
- 备份与归档

## 16. 常见故障处理

### 16.1 数据源连续失败

- 检查网络、源站结构和授权状态
- 触发 Provider 熔断和备用源切换
- 前台保留旧数据并标注时间

### 16.2 AI 响应异常

- 检查模型服务、Prompt 版本和 schema 校验日志
- 必要时切换 fallback 模型
- 失败任务进入重试队列或人工复核

### 16.3 队列积压

- 优先检查 `ai_interactive` 队列
- 暂停低优先级批任务
- 临时扩容 Worker

## 17. 上线检查清单

- 数据库迁移完成
- 健康检查通过
- 核心功能烟雾测试通过
- 监控看板正常
- 告警渠道畅通
- 备份任务启用
- 免责声明和合规文案已上线
